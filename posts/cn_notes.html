6 Socket programming with UDP and TCP

6.1 UDP client implement using Python

from socket import *
serverName = 'hostname'
serverPort = 12000
clientSocket = socket(AF_INET, SOCK_DGRAM) # create UDP socket;TCP:SOCK_STREAM
message = input('Input lowercase sentence:')
clientSocket.sendto(message.encode(), (serverName, serverPort)) # attach server name, port to message; send into socket
modifiedMessage, serverAddress = clientSocket.recvfrom(2048) # read reply date from socket
print(modifiedMessage.decode())
clientSocket.close()

6.2 UDP server implement using Python

from socket import *
serverPort = 12000
serverSocket = socket(AF_INET, SOCK_DGRAM)
serverSocket.bind(('', serverPort))
print('The server is ready to receive')
while True:
    message, clientAddress = serverSocket.recvfrom(2048)
    modifiedMessage = message.decode().upper()
    serverSocket.sendto(modifiedMessage.encode(), clientAddress)

4 Principles of reliable data transfer

built on top of an unreliable network

4.1 The core problem: service abstraction vs. implementation

the fundamental challenge: providing a reliable data transfer service (where data is delivered in order and without error) using an underlying unreliable channel (which may corrupt, lose, or reorder packets)

4.1.1 The interface: reliable data transfer protocol (rdt)

sender & receiver application

4.1.2 Finite state machines (FSMs)

modeling protocols to define sender and receiver states, events (e.g., packet arrival or timeout), and actions (e.g., sending and ACK)

4.2 Reliable data transfer protocol (rdt)

rdt1.0: transfer over a reliable channel (no bit errors or loss of packets)

separate FSMs for sender and receiver

rdt2.0: transfer over a channel with bit errors (may flip bits in packet)

  • detect: checksum
  • recover: stop and wait, sender sends one packet, then waits for receiver's response
    • acknowledgements (ACKs): receiver explicitly tells sender that pkt received OK
    • negative acknowledgements (NAKs): receiver explicitly tells sender that pkt had errors (sender retransmits pkt on receipt of NAK)
    • reliable but performance poor: utilization \(U_{sender}=\frac{L/R}{RTT+L/R}\) is small

corrupted packet scenario:

flaw:

  • ACK/NAK corrupted: sender doesn't know what happened at receiver -> can't just retransmit (possible duplicate)
  • handling duplicates: ACK/NAK corrupted -> sender retransmit + adds sequence number to each pkt -> receiver discards duplicate pkt

rdt2.1: handling garbled ACK/NAKs

rdt2.2: a NAK-free protocol

rdt3.0: transfer over channels with errors and loss

  • timers and timeouts: the sender uses a countdown timer to wait a "reasonable" amount of timer for an ACK. If the timer expires (timeout) before an ACK is received, the sender assumes the packet was lost and retransmits it.
    • permature timeout may causes duplication in retransmiting -> the sequence number handles
  • pipelining for performance: allows multiple "in-flight" , yet-to-be-acknowledged packets
    • range of sequence numbers increased
    • buffering at sender and/or receiver

4.3 Two main approaches to pipelining

1. Go-Back-N (GBN)

  • sender window: the sender is allowed a window of up to N unacknowledged packets
  • cumulative ACK: the receiver sends ACK(n) to indicate that all packets up to and including sequence number n have been correctly received
  • single timer: the sender maintains a timer for the oldest in-flight packet. If this timer expires, the sender retransmits all unacknowledged packets in the window
  • receiver behavior: discards out-of-order packets and resends ACK for the highest in-order seq # packet received so far

2. Selective Repeat (SR)

  • individual ACKs: the receiver sends an ACK for each individual packet received
  • buffering: the receiver buffers out-of-order packets to deliver them in-order to the application later
  • multiple timers: a timer for each unacknowledged packet. On timeouts, retransmits only the specific packet that was lost
  • window dilemma: the window size must be less than or equal to half the sequence number space, or the receiver cannot distinguish between a new packet and a retransmission of an old packet

3.5.2 TCP round trip time, timeout

expiration interval: \(TimeoutInterval = EstimatedRTT + 4 \times DevRTT\)

  • exponential weighted moving average (EWMA): \(EstimatedRTT = (1- \alpha) \times EstimatedRTT + \alpha \times SampleRTT\)
  • EWMA of SampleRTT deviation from EstimatedRTT: \(DevRTT = (1-\beta) \times DevRTT + \beta \times |SampleRTT - EstimatedRTT|\)
  • typical value: \(\alpha\) = 0.125, \(\beta\) = 0.25

timer: for oldest unACKed segment

3.6.2 Approaches towards congestion control

  • end-to-end congestion control:
    • no explicit feedback from network
    • congestion inferred from observed loss, delay
    • approach taken by TCP
  • network-assisted congestion control:
    • routers provide direct feedback to sending/receiving hosts with flows passing through congested router
    • may indicate congestion level or explicitly set sending rate
    • TCP, ECN, ATM, DECbit protocols

3.6.3 TCP congestion control: AIMD

senders can:

  • Additive Increase: increase sending rate until packet loss (congestion) occurs
  • Multiplicative Decrease: then decrease sending rate on loss event

TCP rate: \(cwnd/RTT\) bytes/sec

  • sender limits transmission: \(LastByteSent - LastByteAcked \le cwnd\)
  • cwnd is dynamically adjusted in response to observed network congestion (implementing TCP congestion control)
